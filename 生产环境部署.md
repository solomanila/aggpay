# 生产环境部署计划（AWS 印度区）

## 背景

主要服务印度客户，部署于 AWS **ap-south-1（孟买）**区域。  
架构原则：每个 Java 服务独立一台 VPS，中间件按重要性拆分，全部置于同一 VPC 私网。  
**pay-service 业务最重，采用 2 实例部署，通过 Nacos 注册 + Spring Cloud Gateway 负载均衡，无需额外 LB 组件。**

---

## 服务清单

### Java 服务层（5 台）

| # | 服务 | 端口 | 实例数 | 说明 |
|---|------|------|--------|------|
| 1 | gateway | 8080 | 1 | 统一入口，流量最大 |
| 2 | auth-service | 8081 | 1 | 认证登录 |
| 3a | pay-service（主）| 8084 | **2** | 订单/通道核心业务，多实例负载均衡 |
| 3b | pay-service（副）| 8084 | **2** | 与主实例同权重，Nacos 自动注册 |
| 4 | admin-service | 8085 | 1 | 管理后台 API |

> user-service 暂跳过，待 VPN 功能开发完成后补充。

#### pay-service 多实例负载均衡原理

```
请求链路：Nginx → gateway（lb://pay-service）→ Nacos 服务发现 → pay-service 实例1 / 实例2
```

- 两台 pay-service 启动后**自动向 Nacos 注册**，服务名相同（`pay-service`）
- Spring Cloud Gateway 路由配置 `uri: lb://pay-service`，由 Nacos 负载均衡器按**轮询**分发
- 任意一台宕机，Nacos 心跳超时后自动摘除，gateway 不再路由到该实例
- **无需 AWS ALB / Nginx upstream**，Nacos 自身即为服务发现 + 软负载

### 中间件层（3 台）

| # | 服务 | 拆分建议 | 理由 |
|---|------|---------|------|
| 5 | MySQL 8 | **独立一台** | 数据最重要，不与任何服务共享 |
| 6 | Redis + Nacos | **合并一台** | 两者都极轻量，Nacos 闲置内存 < 300 MB |
| 7 | RocketMQ（namesrv + broker）| **独立一台** | broker 内存消耗大，消息队列不能抢占 |

### 接入层（1 台）

| # | 服务 | 说明 |
|---|------|------|
| 8 | Nginx 前端 | 独立一台，生产环境不能将静态资源压在 gateway 上 |

## 网络规划（VPC 内部隔离）

所有 8 台置于同一 **VPC + 私有子网**，最小化公网暴露面。

### 公网入口（Security Group 开放）

| 端口 | 协议 | 来源 | 说明 |
|------|------|------|------|
| 80 | TCP | 0.0.0.0/0 | Nginx 前端（台8） |
| 443 | TCP | 0.0.0.0/0 | HTTPS（台8，后续配置）|
| 8080 | TCP | 仅限运维 IP | Gateway 调试，上线后关闭 |
| 22 | TCP | 仅限运维 IP | SSH，所有台统一限制 |

### 私网互通（VPC 内，不经公网）

| 端口 | 服务 | 允许访问方 |
|------|------|---------|
| 3306 | MySQL | 5 台 Java 服务 |
| 6379 | Redis | 5 台 Java 服务 |
| 8848 | Nacos | 5 台 Java 服务 |
| 9876 | RocketMQ namesrv | pay-service 实例1、实例2 |
| 10911 | RocketMQ broker | pay-service 实例1、实例2 |

## 采购与部署顺序

```
第1步：创建 VPC + 私有子网 + Security Group（基础网络先建）
第2步：购买并初始化 MySQL 台（其他服务强依赖，最先启动）
第3步：购买并启动 Redis + Nacos 台（Java 服务注册发现依赖）
第4步：购买并启动 RocketMQ 台
第5步：购买 5 台 Java 服务，逐个部署
        顺序：auth → pay实例1 → pay实例2 → admin → gateway
        pay 两台启动后在 Nacos 控制台确认双实例均已注册，再启动 gateway
第6步：购买 Nginx 台，部署前端构建产物
```

## 部署方式

生产环境每台独立启动对应服务，**不使用 docker-compose 全量启动**，改为各台单独运行：

```bash
# gateway 台
docker run -d \
  --name gateway \
  -p 8080:8080 \
  -e SPRING_CLOUD_NACOS_DISCOVERY_SERVER_ADDR=<Nacos私网IP>:8848 \
  -e SPRING_DATA_REDIS_HOST=<Redis私网IP> \
  -e NACOS_SERVER=<Nacos私网IP>:8848 \
  payadmin/gateway:latest

# pay-service 实例1（台3a）
docker run -d \
  --name pay-service \
  -p 8084:8084 \
  -e SPRING_CLOUD_NACOS_DISCOVERY_SERVER_ADDR=<Nacos私网IP>:8848 \
  -e SPRING_DATASOURCE_URL="jdbc:mysql://<MySQL私网IP>:3306/aggpay?useSSL=false&serverTimezone=Asia/Kolkata&allowPublicKeyRetrieval=true" \
  -e SPRING_DATASOURCE_USERNAME=root \
  -e SPRING_DATASOURCE_PASSWORD=<密码> \
  -e ROCKETMQ_NAME_SERVER=<RocketMQ私网IP>:9876 \
  -e NACOS_SERVER=<Nacos私网IP>:8848 \
  payadmin/pay-service:latest

# pay-service 实例2（台3b）—— 命令完全相同，在另一台 VPS 上执行
# 两台使用同一镜像，Nacos 以 IP 区分实例，gateway 自动感知双节点
```

各服务的私网 IP 在 AWS 控制台分配后填入，通过环境变量注入，无需修改 application.yml。

---

## 关键注意事项

1. **Nacos 地址**：gateway / auth-service / pay-service 的 `bootstrap.yml` 若硬编码了 `127.0.0.1:8848`，须改为 `${NACOS_SERVER:127.0.0.1:8848}` 后重新打镜像。

2. **MySQL 数据卷**：EBS 100 GB 挂载至 `/data`，docker volume 映射到 `/data/mysql`，防止实例重启丢数据。

3. **Elastic IP**：gateway 台和 Nginx 台各绑定一个 Elastic IP，重启 IP 不变。

4. **镜像仓库**：建议使用 AWS ECR（ap-south-1）存储 Docker 镜像，各台 pull 速度快且免出流量费。

5. **时区**：所有实例启动时设置 `TZ=Asia/Kolkata`（IST，UTC+5:30）。

## pay-service 扩缩容说明

| 场景 | 操作 |
|------|------|
| 业务增长需要第 3 个实例 | 新增一台 t3.medium，启动同镜像，自动注册 Nacos |
| 某实例异常需下线 | `docker stop pay-service`，Nacos 30s 心跳超时后自动摘除 |
| 滚动更新镜像 | 先停实例2 → 更新 → 启动实例2 → 再停实例1 → 更新 → 启动实例1，全程不停服 |
